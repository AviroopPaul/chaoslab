# ChaosLab — Common System Design Interview Questions: Research Dossier

Purpose: a source-cross-referenced ranking of the most commonly asked system design
interview questions, filtered to the 12 that are faithfully **modelable** in
ChaosLab's simulator (`src/lib/sim/catalog.ts`, `src/lib/sim/types.ts`,
`src/lib/sim/engine.ts`), each written up as a self-contained dossier a downstream
agent can turn directly into a playground question + grading rubric.

## What the simulator can and can't express

Read before using any dossier below. ChaosLab models a **single-region backend
request graph** with 9 component kinds: `users`, `cdn`, `loadbalancer`,
`ratelimiter`, `server` (instances, autoscale, retries, circuit breaker),
`cache` (hit ratio, capacity), `database` (shards, read replicas, max
connections — Little's Law pool cap), `queue` (workers, jobs/worker, queue vs.
pub/sub fan-out), and `storage`. Traffic is a single global `{read, write}`
RPS split (`readWriteRatio`) generated by `users` nodes and flows through a
topologically-ordered graph; the engine reports availability, p50/p99, cost,
and a healthy/degraded/meltdown verdict. Fixed capacity constants worth
knowing for every rubric below: **load balancer 200,000 rps** (per node,
hard cap), **cache 300,000 rps default** (raise via `capacityRps`), **CDN
5,000,000 rps default**, **database 8,000 read rps / 4,000 write rps per
shard**, **queue enqueue ceiling 500,000 rps** (drain = `workers ×
jobsPerWorkerRps`), **storage 100,000 rps**. Cost is computed from the
catalog's `costPerMonth` formulas — flat per configured size for every kind
except CDN, whose cost scales linearly with served RPS (`200 +
servedRps × 0.5`), which several dossiers below call out explicitly since it
dominates budget at streaming/media scale.

There is **no** geo/proximity indexing, no real-time bidirectional
(WebSocket/long-poll) semantics, no operational-transform/CRDT modeling, no
multi-region/replication-lag modeling, and no matching-engine/order-book
semantics. Every dossier below is scoped to stay inside those boundaries; the
[Exclusions](#exclusions) section at the bottom lists top questions that don't
fit and why.

## 1. URL Shortener

**Tags:** caching, sharding, key generation · **Difficulty:** Easy

### Frequency evidence
Alex Xu Vol. 1, ch. 8 ("Design A URL Shortener"); HelloInterview lists Bitly
in its Easy tier; Grokking's curriculum opens its real-problems module with
"URL Shortening Service (TinyURL)"; cited across ByteByteGo/Exponent/dev.to
round-ups as *the* most common cross-company opener question.
([HelloInterview breakdowns](https://www.hellointerview.com/learn/system-design/problem-breakdowns/overview),
[Grokking curriculum](https://www.grokkingsystemdesign.com/curriculum))

### Problem statement
Design a service like Bitly/TinyURL: given a long URL, generate a short,
unique alias and redirect visitors from the short link back to the original
URL with very low latency. Support optional custom aliases and link
expiration. Target scale: **100M DAU**, **100M new links created per day**,
with redirects vastly outnumbering creations (~**100:1 read:write**). Peak
target load: **100,000 rps total** (99,000 read / 1,000 write). Availability
target 99.9%+ — a broken short link is a highly visible failure. Redirect p99
latency target under 300ms.

### Simulation targets
`global.users = 5,000,000`, `rpsPerUser = 0.02`, `readWriteRatio = 0.99` →
100,000 rps offered (99,000 read / 1,000 write).

### Reference architecture
```
users → loadbalancer → server(×N) → cache → database
                            server ────────────→ database   (write path)
                            cache  ────────────→ database   (miss fallback)
```
| Node | Config |
|---|---|
| `loadbalancer` | `algorithm: round-robin` |
| `server` | `instances: 30, rpsPerInstance: 5000` (150,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.92, capacityRps: 300000` (default) — maps `shortCode → longURL` |
| `database` | `shards: 1, readReplicas: 1, maxConnections: 500` — sharded by hash of the short code, not by creation order |

Edges: `users→lb`, `lb→server`, `server→cache` (reads), `server→database`
(writes), `cache→database` (cache-miss fallback).

### Key design insights
- Hashing/base62-encoding the short code (or a distributed counter, e.g. a
  Snowflake-style ID generator) keeps keys quasi-random, so sharding by
  `hash(code)` spreads load evenly — a sequential/auto-increment key would
  hot-spot whichever shard holds the newest IDs.
- The cache is the single highest-leverage component: with a ~100:1 read:write
  ratio, a high hit ratio removes the overwhelming majority of database load
  for a fraction of the cost.
- Write volume is tiny relative to reads, so a synchronous write straight to
  the database is fine — reaching for a queue here is unnecessary complexity
  the load doesn't justify.
- 301 (permanent) vs. 302 (temporary) redirects is a classic follow-up: 301 is
  far more cacheable by browsers/CDNs (lower origin load) but breaks
  per-click analytics, since the browser stops asking your server at all.
- Custom aliases require a uniqueness check before insert — an extra
  read-before-write on the create path, worth naming even though it doesn't
  change the aggregate RPS story much.

### Common mistakes / nuances
- No load balancer / a single server as a SPOF.
- Forgetting the write path entirely — sizing the database only for reads and
  ignoring the uniqueness-check + insert cost of link creation.
- Assuming a 100% cache hit ratio (over-optimistic) or not noticing what
  happens to the ~8% of reads that miss.
- Sharding by sequential/auto-increment ID instead of a hashed key, causing a
  hot shard for all recently created links.
- No abuse protection on the create endpoint (link-spam) — a rate limiter in
  front of the write path is a reasonable addition, though not required at
  this dossier's target scale.

### Grading rubric suggestions
- **Required components:** `loadbalancer`, ≥1 `server`, `cache`, `database`.
- **Topology:** `users→loadbalancer→server`; `server` feeds both `cache`
  (reads) and `database` (writes); `cache` feeds `database`.
- **Config:** `cache.hitRatio ≥ 0.85`; `database.shards ≥ 1` with
  `maxConnections` sized so the pool isn't the binding constraint (Little's
  Law: `maxConnections × 1000/12 ≥ database read+write capacity`).
- **Simulation outcomes at 100,000 rps target load:** `availability ≥ 0.995`;
  `p99Ms < 300`; `costPerMonth ≤ $4,000/mo`.

## 2. API Rate Limiter

**Tags:** admission control, shedding, distributed counters · **Difficulty:** Easy–Medium

### Frequency evidence
Alex Xu Vol. 1, ch. 4 ("Design A Rate Limiter"), one of the earliest and most
frequently cited infra-design chapters; appears in Grokking's fundamentals
module and HelloInterview's Medium tier; ChaosLab's own `ratelimiter`
component is a direct implementation of the token-bucket-style admission gate
this question asks candidates to design.
([HelloInterview breakdowns](https://www.hellointerview.com/learn/system-design/problem-breakdowns/overview))

### Problem statement
Design a rate limiter that sits in front of a large API platform and caps
per-client (or global) request throughput, shedding excess traffic before it
can overload backend services — without adding meaningful latency to admitted
requests, and while staying consistent across many stateless gateway
instances. Scale: a platform serving **200,000 rps** of steady-state API
traffic (read:write ≈ 4:1, i.e., mostly GETs), with occasional bursts up to 2×
that must be shed cleanly rather than cause a backend meltdown.

### Simulation targets
`global.users = 10,000,000`, `rpsPerUser = 0.02`, `readWriteRatio = 0.8` →
200,000 rps offered (160,000 read / 40,000 write). Run a second "stress" pass
at 2× `users` to demonstrate deliberate shedding.

### Reference architecture
```
users → ratelimiter → loadbalancer(×2) → server(×N) → cache → database
                                             server ─────────→ database
                                             cache  ─────────→ database
```
| Node | Config |
|---|---|
| `ratelimiter` | `limitRps: 220000` — sized just above sustained load so it stays dormant at baseline and only sheds during the stress test |
| `loadbalancer` ×2 | `algorithm: least-connections` (a single LB's 200,000 rps cap is below this target, so ≥2 are required) |
| `server` | `instances: 60, rpsPerInstance: 5000` (300,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.85, capacityRps: 300000` |
| `database` | `shards: 12, readReplicas: 1, maxConnections: 3000` |

### Key design insights
- A rate limiter is a **stateless admission gate**, not a queue — it doesn't
  slow down what it admits, it only decides pass/shed at the door. Placing it
  *before* the load balancer protects every downstream tier at once, rather
  than protecting servers but leaving the LB itself exposed.
- Sizing the limit **above** current sustained load (not at the bleeding
  edge) means it acts purely as a safety net in the common case — a
  well-sized limiter should rarely be the thing actually doing the shedding.
- A real distributed rate limiter needs synchronized counters across every
  gateway instance (typically a shared Redis cluster with a sliding-window or
  token-bucket algorithm) — that algorithmic/consistency detail is exactly
  what this component abstracts away; the simulator models the *capacity and
  shedding behavior*, not the counter-synchronization mechanism.
- Layering limiters (a coarse edge/IP-based limiter plus a stricter
  per-API-key limiter closer to the service) is a common real-world follow-up
  even though this dossier's core graph uses a single limiter for clarity.
- Shed traffic (`shedRps`) is deliberately excluded from the availability
  calculation's denominator — a well-designed limiter's rejections are a
  policy decision, not a reliability failure, and the rubric should treat
  them that way.

### Common mistakes / nuances
- No load balancer in front of the limiter/servers, or only one LB at a scale
  that exceeds its fixed 200,000 rps cap.
- Confusing "shed" (deliberate, policy-driven) with "dropped" (an error) when
  narrating the design — candidates should explicitly distinguish them.
- Setting the limit so low it constantly sheds legitimate traffic at normal
  load, rather than sizing it as a safety net above sustained demand.
- Ignoring the write path's tighter downstream constraint (database shard
  count) — a limiter alone doesn't make the backend infinitely scalable.
- Treating the limiter as a full backend service with its own database,
  rather than the fast, stateless, low-latency gate it needs to be.

### Grading rubric suggestions
- **Required components:** `ratelimiter`, `loadbalancer` (≥2 at this scale),
  ≥1 `server`, `cache`, `database`.
- **Topology:** `ratelimiter` precedes `loadbalancer`/`server` (admission
  control before the fleet, not after).
- **Config:** `ratelimiter.limitRps` set explicitly (not left at a default
  that's wildly mismatched to load); `database.shards` sized so write
  capacity (`shards × 4000`) clears the write rps at ≤90% utilization.
- **Simulation outcomes at 200,000 rps target load:** `availability ≥ 0.995`;
  `p99Ms < 200`; `costPerMonth ≤ $15,000/mo`. At 2× stress load, expect
  `shedRps > 0` and `availability` still ≥ 0.99 (shedding, not dropping).

## 3. News Feed / Social Feed

**Tags:** caching, async fan-out, sharding · **Difficulty:** Medium

### Frequency evidence
Alex Xu Vol. 1, ch. 11 ("Design A News Feed System"); Grokking's curriculum
covers both "Instagram" and "Facebook Newsfeed"; HelloInterview lists
Instagram and FB News Feed in its Medium tier; ChaosLab already ships a
tuned "Instagram-Style Social Feed" preset (`src/lib/sim/presets.ts`) built
against the real TAO/Memcache/Haystack papers this dossier reuses.
([HelloInterview breakdowns](https://www.hellointerview.com/learn/system-design/problem-breakdowns/overview))

### Problem statement
Design a social feed (Instagram/Facebook-style): users post text/photo/video
updates and see a ranked, near-real-time feed of updates from people they
follow. Functional requirements: create post, fetch home feed, like/comment.
Scale: **20M concurrent users**, **~95:5 read:write ratio** (feed scrolling
and profile views vastly outnumber posts/likes, consistent with Meta's TAO
paper reporting ~500:1 graph-read amplification), target **1,000,000 rps**
offered load split evenly across a media-delivery lane and a main API lane
(a structural artifact of this simulator's even edge fan-out — see
Simplifications). Feed-read p99 target under 300ms.

### Simulation targets
`global.users = 20,000,000`, `rpsPerUser = 0.05`, `readWriteRatio = 0.95` →
1,000,000 rps offered, split 50/50 by the engine's `users` fan-out into a
500,000 rps CDN/media lane and a 500,000 rps main API lane (475,000 read /
25,000 write each).

### Reference architecture
```
                users
        ┌─────────┴─────────┐
        ▼                   ▼
   cdn (media)        ratelimiter
        │                   │
        ▼                   ▼
    storage         loadbalancer(×3)
                            │
                            ▼
                     server (autoscaled)
                       │        │
                       ▼        ▼
                    cache     queue (fan-out writes)
                       │        │
                       ▼        ▼
                   database   database (feed store)
```
| Node | Config |
|---|---|
| `cdn` | `hitRatio: 0.93, capacityRps: 5000000` — photo/video reads bypass the app tier entirely |
| `storage` | default — origin media store |
| `ratelimiter` | `limitRps: 550000` — safety net above the main-lane's 500,000 rps |
| `loadbalancer` ×3 | `algorithm: least-connections` |
| `server` | `autoscale: on, minInstances: 250, maxInstances: 450, targetUtilization: 0.6, rpsPerInstance: 2000` |
| `cache` | `hitRatio: 0.99, capacityRps: 800000` — Memcached-style object/feed cache |
| `database` (main) | `shards: 2, readReplicas: 0, maxConnections: 800` — social graph/post store, sees only cache-miss reads |
| `queue` | `workers: 400, jobsPerWorkerRps: 100, mode: queue` — async fan-out (Celery/Kafka-class) |
| `database` (feed store) | `shards: 10, readReplicas: 1, maxConnections: 2500` — precomputed per-user feed, written by the queue |

### Key design insights
- A social feed is fundamentally a **read-amplification problem**: every
  layer (CDN for media, in-memory cache for graph/post reads) exists purely
  to keep reads off the database, and any erosion in hit ratio multiplies
  straight through to database load.
- Sharding the post/graph store lets any application server route a request
  to the right shard directly (Instagram famously encodes the shard ID inside
  the photo ID itself) — sharding here is as much about avoiding a
  coordination/lookup bottleneck as it is about raw capacity.
- Publishing never blocks on delivering to followers: fan-out into
  precomputed per-user feeds happens asynchronously via a queue — this is
  exactly why celebrity/huge-follower-count accounts need a different,
  pull-based read path in the real system (a classic follow-up, not modeled
  here).
- A single load balancer's fixed 200,000 rps cap is the first thing to give
  out at this scale — this dossier requires **at least 3** load balancer
  nodes, a good rubric discriminator.
- CDN cost in this engine scales linearly with served throughput
  (`$0.5/rps/mo`), not with configured size — at social-media media-serving
  scale this becomes the dominant cost line, mirroring how CDN/egress
  genuinely dominates real infrastructure spend for image/video-heavy
  platforms.

### Common mistakes / nuances
- Forgetting the async fan-out queue and writing directly to the feed store
  synchronously on every post (ties post-creation latency to fan-out cost).
- A single load balancer node at this traffic level (200,000 rps hard cap
  vs. a 500,000 rps lane).
- Treating cache hit ratio as free/fixed — not connecting a hit-ratio drop to
  a proportional database load spike.
- No distinction between the durable social-graph/post store and the
  separately-scaled, queue-fed feed store — collapsing them into one
  database understates the fan-out problem.
- Ignoring celebrity/hot-key skew: this design assumes uniform fan-out cost
  per post, which breaks down for accounts with millions of followers (an
  explicit simplification, not a rubric requirement).

### Grading rubric suggestions
- **Required components:** `cdn`, `storage`, `loadbalancer` (≥3),
  `ratelimiter`, `server`, `cache`, `database` (≥2 distinct database nodes),
  `queue`.
- **Topology:** `users` feeds both `cdn` and the `loadbalancer` lane; `cdn`
  feeds `storage`; `server` feeds `cache` (reads) and `queue` (writes);
  `queue` feeds the feed-store `database`; `cache` feeds the main `database`.
- **Config:** `cache.hitRatio ≥ 0.95`; `queue.workers × jobsPerWorkerRps ≥
  1.2 × write rps`; feed-store `database.shards` sized so write capacity
  clears queue-drained demand at ≤85% utilization.
- **Simulation outcomes at 1,000,000 rps target load:** `availability ≥
  0.99`; `p99Ms < 400`; `costPerMonth ≤ $350,000/mo` (CDN's linear
  servedRps-based cost dominates this budget by design — flag this as
  expected, not a red flag, and discuss it as a design trade-off).

### Simplifications
Inherited from ChaosLab's own Instagram preset: `users` fans out evenly
across its outgoing edges, so the CDN/media lane and main API lane each get
exactly 50% of total offered traffic regardless of the real ~95/5
media-dominant skew — size targets accordingly (double the desired per-lane
load to get the correct `users`/`rpsPerUser` input). TAO's graph-aware
semantics and celebrity push/pull hybrid fan-out are not modeled.

## 4. Chat System

**Tags:** async writes, sharding, HTTP-modeled messaging · **Difficulty:** Medium–Hard

### Frequency evidence
Alex Xu Vol. 1, ch. 12 ("Design A Chat System"); Grokking's curriculum
includes "Facebook Messenger"; community signal notes chat/messaging design
has been asked at Meta for 5+ years running.
([Blind — master question list](https://www.teamblind.com/post/master-list-of-all-system-design-interview-questions-w34wkv2r))

### Problem statement
Design a messaging backend (WhatsApp/Messenger/Slack-class): support 1:1 and
group conversations, sending messages, and fetching/syncing conversation
history. **Scope note:** this simulator has no WebSocket/long-poll/real-time
push semantics, so push-delivery, presence, and typing indicators are
explicitly out of scope — model only the send (write) and fetch/sync (read)
HTTP-shaped API traffic. Scale: **100M DAU**, ~40 messages sent+received per
user per day, producing roughly **50,000 rps** of combined send/fetch
traffic, with fetch/sync traffic somewhat exceeding sends (**read:write ≈
7:3**).

### Simulation targets
`global.users = 100,000,000` sized against `rpsPerUser` and `readWriteRatio =
0.7` such that offered load is 50,000 rps (35,000 read / 15,000 write) — e.g.
`rpsPerUser ≈ 0.0005`.

### Reference architecture
```
users → loadbalancer → server → cache → database
                          server → queue → database
```
| Node | Config |
|---|---|
| `loadbalancer` | `algorithm: round-robin` |
| `server` | `instances: 50, rpsPerInstance: 1500` (75,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.9, capacityRps: 300000` — recent per-conversation message history |
| `queue` | `workers: 250, jobsPerWorkerRps: 100, mode: queue` — durable async message persistence |
| `database` | `shards: 6, readReplicas: 1, maxConnections: 1500` — message store, **sharded by conversation_id** |

### Key design insights
- Shard the message store by **conversation_id**, not `user_id` — this keeps
  both participants' full history colocated for a fast single-shard fetch;
  sharding by `user_id` instead would scatter a group conversation's history
  across many shards and require a fan-in read on every fetch.
- Persisting a sent message asynchronously via a durable queue improves
  perceived send latency (the client can be told "sent" once the message is
  durably enqueued) — but the queue must be a durable, replicated log
  (Kafka-class), not an in-memory buffer, or a crash before drain loses
  messages.
- A cache in front of the message store absorbs the read-heavy "catch up on
  my open conversations" traffic — the sync/fetch pattern is comparatively
  read-heavy relative to sends.
- Group-chat fan-out is structurally a pub/sub problem (one send, many
  recipients) — toggling the queue's `mode` to `pubsub` and setting
  `subscriberCount` to a group's member count is a good hands-on way to see
  fan-out cost scale with group size, even though the core dossier graph uses
  plain queue mode for simplicity.
- Presence, delivery receipts, and typing indicators are a separate,
  much-higher-QPS, ephemeral system that typically does **not** live in the
  durable message store — worth naming explicitly as out of scope.

### Common mistakes / nuances
- Modeling chat as purely synchronous with no queue, letting send-path p99
  be dominated directly by database write latency.
- Sharding by `user_id` instead of `conversation_id`, causing cross-shard
  reads for every conversation fetch.
- Treating 1:1 and group chat identically — group fan-out multiplies write
  amplification by group size, which plain queue mode doesn't capture (use
  pub/sub mode to illustrate it).
- No cache for "which conversations are currently open," hammering the
  database with repeat catch-up reads.
- Treating queue backlog as harmless — an undrained backlog is silent
  message-delivery delay, not a free buffer.

### Grading rubric suggestions
- **Required components:** `loadbalancer`, `server`, `queue`, `database`,
  `cache`.
- **Topology:** `server` feeds `cache` (reads) and `queue` (writes); `queue`
  feeds `database`; `cache` feeds `database`.
- **Config:** `queue.workers × jobsPerWorkerRps ≥ 1.2 × write rps`;
  `database.shards` sized so write capacity clears queue-drained demand;
  `cache.hitRatio ≥ 0.85`.
- **Simulation outcomes at 50,000 rps target load:** `availability ≥ 0.99`;
  `p99Ms < 250`; `costPerMonth ≤ $20,000/mo`.

## 5. Video Streaming

**Tags:** CDN economics, async transcoding pipeline · **Difficulty:** Hard

### Frequency evidence
Grokking's curriculum includes "YouTube or Netflix"; HelloInterview lists
YouTube in both its Medium tier (basic) and Hard tier ("YouTube Top K");
ByteByteGo's course deep-dives YouTube design; ChaosLab already ships a
tuned "Netflix-Style Streaming" preset built against Netflix's own TechBlog
sources, extended here with an upload/transcoding pipeline the existing
preset (playback-only) doesn't cover.
([Netflix TechBlog — Open Connect](https://blog.apnic.net/2018/06/20/netflix-content-distribution-through-open-connect/),
[Netflix TechBlog — EVCache](https://netflixtechblog.com/announcing-evcache-distributed-in-memory-datastore-for-cloud-c26a698c27f7))

### Problem statement
Design a video platform (YouTube/Netflix-class): users upload videos (which
must be transcoded into multiple resolutions before being watchable), browse
metadata, and stream video. **Scope note:** the actual video-byte delivery
path is dominated by CDN capacity, not app-server logic — model it as a
direct CDN→origin-storage lane bypassing the app tier, per the existing
preset's approach. Scale: **200M DAU**, view traffic vastly exceeds uploads
(**read:write ≈ 98:2**), target **600,000 rps offered**, split 50/50 by the
engine's fan-out into a 300,000 rps video-byte lane and a 300,000 rps
metadata/control-plane lane.

### Simulation targets
`global.users = 30,000,000`, `rpsPerUser = 0.02`, `readWriteRatio = 0.98` →
600,000 rps offered, split into two 300,000 rps lanes (294,000 read / 6,000
write each).

### Reference architecture
```
                     users
             ┌─────────┴─────────┐
             ▼                   ▼
        cdn (video)         ratelimiter
             │                   │
             ▼                   ▼
          storage         loadbalancer(×2)
                                  │
                                  ▼
                           server (autoscaled)
                             │        │
                             ▼        ▼
                          cache     queue (transcoding)
                             │        │
                             ▼        ▼
                          database ◄──┘
```
| Node | Config |
|---|---|
| `cdn` | `hitRatio: 0.95, capacityRps: 10000000` |
| `storage` | default — origin video store |
| `ratelimiter` | `limitRps: 350000` |
| `loadbalancer` ×2 | `algorithm: least-connections` |
| `server` | `autoscale: on, minInstances: 400, maxInstances: 600, targetUtilization: 0.6, rpsPerInstance: 1000` |
| `cache` | `hitRatio: 0.9, capacityRps: 500000` — video metadata (views/likes/comments) |
| `queue` | `workers: 100, jobsPerWorkerRps: 100, mode: queue` — transcoding pipeline |
| `database` | `shards: 3, readReplicas: 1, maxConnections: 800` — video metadata store, fed by both cache-miss reads and transcoding-complete writes |

### Key design insights
- Video bytes and control-plane metadata are fundamentally different traffic
  classes: video should bypass the app tier entirely via a CDN straight to
  origin storage, never consuming API-server capacity.
- Upload processing (transcoding into multiple resolutions) is asynchronous
  and queue-driven — a user's upload "succeeds" once durably enqueued, not
  once every rendition finishes encoding, which can take minutes.
- CDN hit ratio is existential at this scale: with this engine's linear
  `$0.5/rps/mo` CDN cost model, losing hit ratio doesn't just add latency —
  it visibly raises served-throughput cost, since more reads pass through
  uncached (though note the cost formula charges on *served*, not *missed*,
  RPS — a modeling simplification worth naming).
- Metadata (view counts, comments) is cache-friendly and read-heavy, the same
  lesson as the news-feed dossier — but the write side here is dominated by
  async transcoding-completion events, not direct user writes.
- Multi-region CDN placement, adaptive bitrate selection, and per-title
  regional caching are real considerations this single-region simulator
  can't express.

### Common mistakes / nuances
- Routing video bytes through the app-server tier, massively
  over-provisioning servers for work they should never see.
- Treating "upload" as a single synchronous write, blocking the user on a
  potentially minutes-long transcode.
- A single load balancer at this scale (300,000 rps lane vs. a 200,000 rps
  per-node cap) — needs ≥2.
- Assuming near-100% CDN hit ratio without justifying it — new/unpopular
  videos have much lower cache freshness.
- Ignoring that CDN cost, in this engine, scales with the throughput it
  serves — a design review should explicitly weigh hit-ratio investment
  against this line item.

### Grading rubric suggestions
- **Required components:** `cdn`, `storage`, `loadbalancer` (≥2),
  `ratelimiter`, `server`, `cache`, `database`, `queue`.
- **Topology:** `users` feeds both `cdn` and the `loadbalancer` lane; `cdn`
  feeds `storage`; `server` feeds `cache` (reads) and `queue` (writes);
  `queue` feeds `database`; `cache` feeds `database`.
- **Config:** `cdn.hitRatio ≥ 0.9`; `queue.workers × jobsPerWorkerRps ≥ 1.2 ×
  upload write rps`; `database.shards` sized so both read and write caps
  clear demand at ≤90% utilization.
- **Simulation outcomes at 600,000 rps target load:** `availability ≥ 0.99`;
  `p99Ms < 400` (metadata calls, not the video stream itself);
  `costPerMonth ≤ $220,000/mo` (CDN's linear cost dominates — call this out
  as the expected, realistic outcome for a video platform, not a design
  flaw).

### Simplifications
Same `users`-fan-out caveat as the news-feed dossier: the CDN/video lane and
metadata lane each receive exactly 50% of offered traffic in this engine,
not the real ~95%+/5% video-dominant skew.

## 6. Notification System

**Tags:** pub/sub fan-out, multi-channel delivery · **Difficulty:** Medium

### Frequency evidence
Alex Xu Vol. 1, ch. 10 ("Design A Notification System"); Grokking Vol. II
includes a dedicated "distributed notification service" module; this
dossier is also the clearest showcase of ChaosLab's `queue` component's
`mode: 'pubsub'` knob, which exists specifically for this shape of problem.

### Problem statement
Design a notification system that delivers events (e.g., "someone liked your
post", "your order shipped") across multiple channels (push, email, SMS) to
potentially millions of recipients, without blocking the triggering request
on slow third-party providers (APNs/FCM/Twilio/SES), and without spamming
users with duplicate/excessive notifications. Scale: **500M registered
users**, peak trigger-event rate **35,000 writes/sec**, plus **15,000 rps**
of users fetching their in-app notification inbox (**read:write ≈ 3:7** —
this is a write-dominated problem, unlike most of this dossier set).

### Simulation targets
`global.users` and `rpsPerUser` sized to 50,000 rps total, `readWriteRatio =
0.3` → 15,000 read / 35,000 write.

### Reference architecture
```
users → loadbalancer → server → cache → database
                          server → queue (pub/sub, 3 subscribers) → database
```
| Node | Config |
|---|---|
| `loadbalancer` | default |
| `server` | `instances: 75, rpsPerInstance: 1000` (75,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.9, capacityRps: 300000` — inbox reads + recent-send dedup keys |
| `database` (log/prefs) | `shards: 1, readReplicas: 0, maxConnections: 400` — cache-miss inbox reads only |
| `queue` | `workers: 1600, jobsPerWorkerRps: 100, mode: pubsub, subscriberCount: 3` — one logical subscriber per channel (push/email/SMS) |
| `database` (delivery status) | `shards: 13, readReplicas: 0, maxConnections: 2000` |

### Key design insights
- **Pub/sub, not plain queue mode, is the correct shape**: one triggering
  event must reach *multiple independent consumers* (push/email/SMS channel
  workers), each needing its own full copy of the stream — a plain queue
  would load-split the work, so only one channel would ever see any given
  message.
- Because every subscriber adds its own full drain burden, adding channels
  **multiplies total consumption work**, not just message volume — workers
  must be sized for `event rate × subscriber count`, a classic
  under-provisioning trap.
- Per-user/per-notification-type dedup and rate limiting (don't send 50
  "someone liked your post" pings in 10 seconds) belongs in the fast cache
  layer with short TTLs, not the durable log.
- Provider unavailability (a slow APNs/FCM/SMS gateway) must never block the
  triggering request — hence async pub/sub dispatch, not a synchronous call
  from the API server.
- Priority tiers (critical account-security alerts vs. marketing blasts)
  argue for separate topics/queues with independent worker pools in a real
  system — a single shared queue risks a marketing blast starving critical
  alerts (not modeled in this dossier's core graph, but a strong follow-up).

### Common mistakes / nuances
- Calling third-party providers synchronously from the API server, tying
  user-facing latency/availability to APNs/FCM/Twilio's own reliability.
- Using plain queue mode instead of pub/sub when multiple channels must all
  receive the same event — silently under-delivers to all but one channel.
- No dedup/rate-limit layer, leading to notification storms.
- Sizing workers for the raw event rate instead of `event rate ×
  subscriberCount` in pub/sub mode.
- No separate delivery-status tracking — no way to tell whether a
  notification actually landed.

### Grading rubric suggestions
- **Required components:** `loadbalancer`, `server`, `cache`, `database`
  (≥1), `queue`.
- **Topology:** `server` feeds `cache` (reads) and `queue` (writes); `queue`
  feeds `database`.
- **Config:** `queue.mode == 'pubsub'` **and** `subscriberCount ≥ 3`
  (the single highest-value check for this problem);
  `queue.workers × jobsPerWorkerRps ≥ 1.2 × write rps × subscriberCount`;
  `cache.hitRatio ≥ 0.85`.
- **Simulation outcomes at 50,000 rps target load:** `availability ≥ 0.99`;
  `p99Ms < 300`; `costPerMonth ≤ $85,000/mo` (the pub/sub worker fleet's
  fanout-driven cost dominates — expected and worth naming explicitly).

## 7. Flash-Sale / Ticket Booking

**Tags:** admission control, write serialization, inventory · **Difficulty:** Hard

### Frequency evidence
HelloInterview's "Ticketmaster" is consistently one of their most-practiced
Medium-tier problems; Grokking Vol. II has a dedicated "flash-sale system"
module; commerce/inventory-burst questions appear across dev.to and Exponent
round-ups as a distinct hard-mode category from generic e-commerce.

### Problem statement
Design a high-demand ticket/inventory sale system (e.g., a popular concert
on-sale or a flash deal): thousands of users compete for a strictly limited
number of items simultaneously, the system must **never oversell** (each
seat/item sold exactly once), and it must give fast, honest feedback (booked
or sold-out) even under a massive instantaneous burst. Scale: a burst of
**2M concurrent users** attempting to buy from **50,000 available seats**
within the first minute of an on-sale. Target burst load: **350,000 rps**
total, **write-dominated** (**read:write ≈ 40:60** — purchase attempts, not
browsing, dominate during the burst).

### Simulation targets
`global.users` and `rpsPerUser` sized to 350,000 rps total, `readWriteRatio =
0.4` → 140,000 read / 210,000 write.

### Reference architecture
```
users → ratelimiter (waiting room) → loadbalancer(×2) → server → cache
                                                            server → queue (reservation) → database
```
| Node | Config |
|---|---|
| `ratelimiter` | `limitRps: 250000` — a deliberate **virtual waiting room**, sized well below the raw 350,000 rps burst so overflow is shed cleanly at the edge instead of piling into checkout |
| `loadbalancer` ×2 | `algorithm: least-connections` |
| `server` | `instances: 380, rpsPerInstance: 1000` (380,000 rps cap, ~66% util) |
| `cache` | `hitRatio: 0.7, capacityRps: 300000` — event/venue metadata; deliberately **not** used for live seat counts |
| `queue` | `workers: 430, jobsPerWorkerRps: 500` (215,000 rps drain) — serializes purchase attempts against inventory |
| `database` | `shards: 55, readReplicas: 0, maxConnections: 8000` — inventory + orders, sharded by `event_id` |

### Key design insights
- A flash sale is fundamentally an **admission-control + serialization**
  problem, not a caching problem — you cannot cache your way out of "don't
  oversell the last 5 seats."
- The rate limiter here is a **virtual waiting room**: deliberately shedding
  excess demand at the edge is a *feature*, not a failure — it protects the
  checkout path and gives users fast, honest feedback instead of long hangs
  or timeouts.
- The seat/inventory decrement must be atomic and serialized per item (a
  reservation queue, DB row-level locking, or a distributed lock) — parallel,
  unsynchronized writes are exactly how systems oversell.
- "Seats remaining" is a **poor cache candidate** — it changes on every
  write. Cache the largely-static event/venue/seat-map metadata instead, not
  the live count (this dossier's cache hit ratio is deliberately modest for
  exactly this reason — contrast with the autocomplete dossier's
  intentionally high hit ratio).
- Sharding inventory by `event_id` confines one popular event's write storm
  to its own shard, but a single mega-event with no further partition key can
  still saturate one shard — the reservation queue, not the shard count
  alone, is what protects the database in that worst case.

### Common mistakes / nuances
- No rate limiter/waiting room at all — letting an unbounded burst hit the
  checkout API directly is close to a guaranteed meltdown.
- Caching the live seat count as if it were a normal read-heavy value,
  causing overselling from stale reads.
- A single unsharded, unqueued inventory table under a write-attempt storm.
- Treating "shed" (politely rejected at the waiting room) the same as
  "dropped" (errored) in the design narrative — the UX contract should be
  explicit about which one is happening.
- Forgetting that queue backlog during the burst means a delayed
  confirmed/sold-out response — the design should state that latency
  trade-off explicitly.

### Grading rubric suggestions
- **Required components:** `ratelimiter`, `loadbalancer` (≥2), `server`,
  `cache`, `queue`, `database`.
- **Topology:** `ratelimiter` precedes `loadbalancer`/`server` (admission
  control before checkout); `server` feeds `cache` (reads) and `queue`
  (writes); `queue` feeds `database`.
- **Config:** `ratelimiter.limitRps` set explicitly **below** the raw
  peak-burst rps (demonstrating deliberate shedding, not an accidental
  bottleneck); `queue.workers × jobsPerWorkerRps` sized close to (not wildly
  over) the admitted write rate, reflecting deliberate serialization;
  `database.shards` sized so write capacity clears queue-drained demand at
  ≤90% utilization.
- **Simulation outcomes at 350,000 rps burst load:** total `availability
  ≥ 0.98` (note: `shedRps > 0` is *expected and correct* here, unlike every
  other dossier in this set); `p99Ms < 1000` (booking confirmation can be
  slower than a typical read); `costPerMonth ≤ $70,000/mo` (explicitly a
  burst-provisioned cost — note that it should scale down between sales).

## 8. File Storage / Cloud Drive

**Tags:** object storage, chunking/dedup, async processing · **Difficulty:** Medium

### Frequency evidence
Alex Xu Vol. 2, ch. 9 ("S3-like Object Storage"); HelloInterview lists
Dropbox in its Easy tier; Grokking's curriculum includes "Dropbox" (file
synchronization and distributed storage); a cross-source staple
(dev.to's top-50 list, Exponent).

### Problem statement
Design a cloud file storage/sync service (Dropbox/Google Drive-class):
upload, download, and list files and folders; sync changes across a user's
devices; support large files via chunking and dedup identical content;
share files/folders with permissions. Scale: **50M DAU**, target
**80,000 rps** total (**read:write ≈ 80:20** — downloads/listings vs.
uploads/modifications).

### Simulation targets
`global.users` and `rpsPerUser` sized to 80,000 rps total, `readWriteRatio =
0.8` → 64,000 read / 16,000 write.

### Reference architecture
```
users → loadbalancer → server → cache → database
                          │       (10% of reads auto-route to storage — see note)
                          └→ storage
                          server → queue (chunk processing) → storage
```
| Node | Config |
|---|---|
| `loadbalancer` | default |
| `server` | `instances: 120, rpsPerInstance: 1000` (120,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.85, capacityRps: 300000` — file/folder metadata & permissions |
| `database` | `shards: 2, readReplicas: 1, maxConnections: 600` — metadata store, **sharded by user_id** |
| `queue` | `workers: 250, jobsPerWorkerRps: 100` — async chunk hashing/dedup/virus-scan/thumbnailing |
| `storage` | default — content-addressed, chunked blob store, fed both directly (raw file-byte reads) and via the queue (processed upload commits) |

### Key design insights
- Splitting the **control plane** (metadata, permissions — small,
  latency-sensitive, cacheable) from the **data plane** (actual file bytes —
  huge, infrequently re-read, durable) is the single most important
  decision here — it maps directly onto this engine's built-in rule that
  10% of a server's read traffic automatically routes to object storage
  (static-asset requests) rather than the cache/database.
- Content-addressable, chunked storage (hash each fixed-size chunk, store
  once, reference many times) gives free deduplication across users and
  enables resumable/partial uploads.
- Upload processing (hashing, virus scan, thumbnailing, indexing) is
  asynchronous and queue-driven, so the client's "upload complete" doesn't
  block on all of that work.
- Multi-device sync needs a per-file/per-user change log (version vector or
  logical clock) to detect and resolve conflicts — a data-modeling problem
  this simulator can't express, called out as a simplification rather than a
  rubric requirement.
- Sharding metadata by `user_id` keeps one user's whole file tree on a
  single shard (fast "list my files"), at the cost of a hot-shard risk for a
  small number of very heavy users — the classic sharding-key trade-off.

### Common mistakes / nuances
- Storing file bytes *inside* the metadata database — blows up DB size/cost
  and defeats the whole reason for sharding metadata separately.
- Treating upload as fully synchronous, blocking the client on hashing,
  virus-scanning, and thumbnailing of potentially huge files.
- No dedup — storing byte-identical chunks repeatedly across users.
- Ignoring the sync/conflict-resolution problem entirely — bidirectional
  sync has very different failure modes than a plain upload/download
  service.
- Forgetting metadata-cache invalidation on rename/move/delete, leaving
  stale folder listings.

### Grading rubric suggestions
- **Required components:** `loadbalancer`, `server`, `cache`, `database`,
  `queue`, `storage`.
- **Topology:** `server` feeds `cache` (reads), `queue` (writes), and
  `storage` (direct file-byte reads); `cache` feeds `database`; `queue`
  feeds `storage`.
- **Config:** `cache.hitRatio ≥ 0.8`; `queue.workers × jobsPerWorkerRps ≥
  1.2 × write rps`; `database.shards ≥ 2`.
- **Simulation outcomes at 80,000 rps target load:** `availability ≥ 0.99`;
  `p99Ms < 500` (chunk processing can be slower than pure metadata calls);
  `costPerMonth ≤ $25,000/mo`.

## 9. Web Crawler

**Tags:** rate limiting, dedup, async pipeline · **Difficulty:** Hard

### Frequency evidence
Alex Xu Vol. 1, ch. 9 ("Design A Web Crawler"); Grokking and HelloInterview
both list web crawler design in their Hard tier; recurs across dev.to/
Exponent round-ups as a distinctive "pipeline, not request/response" problem.

### Problem statement
Design a web crawler that fetches and indexes a large portion of the public
web: respect per-domain politeness (robots.txt, crawl-delay), avoid
re-crawling duplicate/unchanged content, extract page content and links to
feed the next crawl frontier, and prioritize re-crawl frequency by page
importance/change rate. **Modeling note:** this problem has no traditional
end-users — reinterpret the `users` node as the crawl-worker fleet issuing
fetch requests. Scale: target crawling **1 billion pages over 30 days**
(~400 pages/sec sustained fetch rate), with each successful fetch producing
roughly one downstream write (extracted content + links), i.e. **read:write
≈ 1:1** at **800 rps** total.

### Simulation targets
`global.users = 8,000` (crawl workers), `rpsPerUser = 0.1`, `readWriteRatio =
0.5` → 800 rps offered (400 read / 400 write).

### Reference architecture
```
users (crawl workers) → ratelimiter (politeness) → server (fetchers) → cache (dedup)
                                                            server → queue (extraction) → database
```
| Node | Config |
|---|---|
| `ratelimiter` | `limitRps: 700` — an aggregate stand-in for thousands of independent per-domain politeness budgets (crawl-delay/robots.txt) |
| `server` | `instances: 25, rpsPerInstance: 50` (1,250 rps cap) — deliberately low per-instance throughput, reflecting network-I/O-bound fetch workers, not CPU-bound app logic |
| `cache` | `hitRatio: 0.6, capacityRps: 300000` — URL/content-hash "already seen" dedup check |
| `queue` | `workers: 60, jobsPerWorkerRps: 10` (600 rps drain) — HTML parsing, link extraction, indexing |
| `database` | `shards: 1, readReplicas: 1, maxConnections: 500` — crawled page store + link graph |

### Key design insights
- The URL/content-hash dedup cache is the highest-leverage component, but it
  has a **fundamentally lower target hit ratio** than a typical read cache
  (this dossier uses ~0.6): you *want* to keep discovering new URLs, so an
  unrealistically high hit ratio here (0.9+) is actually a sign the crawl
  frontier has gone stale, not a sign of good caching — a deliberate contrast
  with the autocomplete dossier's high-hit-ratio design.
- Per-domain politeness naturally maps to the `ratelimiter` component, but a
  real crawler needs *thousands* of independent per-domain budgets, which
  this single-region simulator collapses into one aggregate limiter — an
  explicit, important simplification to name.
- Fetching (network-bound, highly parallel, low CPU per request) and
  extraction/indexing (CPU-bound HTML/text processing) are different
  bottleneck shapes and belong in separate tiers — hence a fetcher fleet
  (`server`) feeding a distinct extraction pipeline (`queue`), not one
  monolith.
- Content-hashing (not just URL) lets you skip re-indexing pages that
  haven't actually changed even when re-fetched — a classic follow-up on the
  dedup cache's design.
- The crawl frontier's prioritization policy (what to fetch next, ordered by
  importance/freshness) is an algorithm/data-structure problem, not a
  capacity problem — this simulator can express the pipeline's *throughput*
  but not its scheduling policy.

### Common mistakes / nuances
- No rate limiting/politeness control at all — a real crawler without this
  gets IP-banned by every site it touches.
- Treating fetch and extraction as one synchronous step, so a
  slow/parse-heavy page blocks the fetcher from moving on.
- No dedup cache — repeatedly re-crawling/re-processing the same URLs,
  wasting the whole pipeline's capacity.
- A single unsharded page-store database backing a billion-page crawl.
- Ignoring storage growth entirely — a billion pages of raw HTML is
  substantial durable storage, worth at least naming even if not wired into
  the core rubric graph.

### Grading rubric suggestions
- **Required components:** `ratelimiter`, `server`, `cache`, `queue`,
  `database`.
- **Topology:** `ratelimiter` precedes `server` (politeness gate before
  fetch); `server` feeds `cache` (dedup-check reads) and `queue`
  (extraction writes); `queue` feeds `database`.
- **Config:** `ratelimiter.limitRps` present and below a naive
  no-politeness fetch rate; `cache.hitRatio` in a **modest** band (0.4–0.75)
  — flag as a design smell if set unrealistically high (≥0.9), since that
  implies the crawler has mostly stopped discovering new content;
  `queue.workers × jobsPerWorkerRps ≥ 1.2 × extraction write rps`.
- **Simulation outcomes at 800 rps target load:** `availability ≥ 0.97`
  (crawlers can tolerate more shedding/backpressure than user-facing
  systems); `p99Ms < 2000` (an inherently background/batch-ish workload);
  `costPerMonth ≤ $6,000/mo`.

## 10. Autocomplete / Typeahead

**Tags:** precomputation, cache design, latency-bound · **Difficulty:** Medium

### Frequency evidence
Grokking's curriculum includes a dedicated "Typeahead Suggestion" lesson
("search query completion with latency requirements"); a recurring
component of search-heavy product design questions across HelloInterview
and Exponent round-ups.

### Problem statement
Design a search-box autocomplete service: as a user types, return the top-K
most likely completions within a very tight latency budget, informed by
overall and trending query popularity, without running a full ranking query
on every keystroke. Scale: **500M DAU**, each issuing several searches/day
with multiple keystroke-level lookups per search, producing **300,000 rps**
of suggestion lookups (read) and a much smaller **3,000 rps** trickle of
"log this completed search" writes feeding an offline aggregation job
(**read:write ≈ 99:1**). Latency target: **p99 under 100ms** — the tightest
bar of any dossier in this set, since every keystroke must feel instant.

### Simulation targets
`global.users` and `rpsPerUser` sized to 303,000 rps total, `readWriteRatio =
0.99` → 300,000 read / 3,000 write.

### Reference architecture
```
users → loadbalancer(×3) → server → cache → database
                              server → queue (query-log ingestion) → database
```
| Node | Config |
|---|---|
| `loadbalancer` ×3 | `algorithm: least-connections` |
| `server` | `instances: 450, rpsPerInstance: 1000` (450,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.97, capacityRps: 500000` — precomputed top-K suggestions per prefix |
| `queue` | `workers: 50, jobsPerWorkerRps: 100` — async query-log ingestion feeding trending recompute |
| `database` | `shards: 1, readReplicas: 1, maxConnections: 400` (default) — suggestion index + trending aggregates |

*Optional addition:* a `cdn` node between `users` and the `loadbalancer`
(`hitRatio ≈ 0.5`) can shave edge latency for the handful of extremely hot,
near-universal prefixes ("a", "the", "how"). It's **not required** in the
core rubric: this engine's linear CDN cost model (`$0.5/rps/mo` on *served*,
not just cached, traffic) makes it an explicit cost/latency trade-off to
discuss rather than a free win — at this dossier's scale it alone would add
roughly $77,000/mo.

### Key design insights
- The defining trick is **precomputation**: at this request volume you
  cannot afford to rank candidates live on every keystroke, so an
  offline/async pipeline (queue-fed) periodically rebuilds a small,
  denormalized "top-K suggestions per prefix" structure that the online path
  only ever *reads*.
- This makes the prefix cache's hit ratio legitimately very high (0.97 here)
  — a deliberate contrast with the web-crawler dossier's dedup cache and the
  flash-sale dossier's live-counter cache, both of which are *structurally*
  low-hit-ratio by design. A good rubric should reward candidates who
  recognize which kind of cache they're building.
- The write path here is really a **logging/telemetry pipeline** (what did
  people search for) feeding an entirely separate, async, lower-urgency job —
  not a user-facing write, which is why it's such a tiny fraction of total
  traffic.
- Trending/personalization (recent spikes, per-user history) pulls the
  design toward a hybrid: a mostly-static base index plus a
  fast-refreshing "trending now" overlay — a common, unmodeled follow-up.
- Unlike almost every other dossier here, the binding design constraint is
  **latency**, not raw throughput headroom — p99 is the number that matters
  most.

### Common mistakes / nuances
- Querying a live full-text search index/database on every keystroke instead
  of precomputing suggestions — guaranteed to blow the latency budget at
  this request volume.
- Setting the prefix cache's hit ratio too low, as if it were a generic
  unpredictable cache — missing that this workload is uniquely cache-
  friendly.
- Conflating the "log this search" write path with the "serve a suggestion"
  read path as if they need the same latency/consistency guarantees.
- Ignoring staleness — an index that's never refreshed misses trending
  queries; the design should name a refresh cadence even conceptually.
- Adding a CDN reflexively without weighing its cost against the (modest)
  latency win, given this engine's linear served-RPS CDN pricing.

### Grading rubric suggestions
- **Required components:** `loadbalancer` (≥3 at this scale), `server`,
  `cache`, `database`, `queue`. (`cdn` optional/bonus.)
- **Topology:** `server` feeds `cache` (reads) and `queue` (writes); `queue`
  feeds `database`; `cache` feeds `database`.
- **Config:** `cache.hitRatio ≥ 0.9` (flag designs below ~0.85 as
  under-leveraging precomputation); `queue.workers × jobsPerWorkerRps ≥ 1.2
  × write rps` without being wildly over-provisioned relative to the tiny
  write volume.
- **Simulation outcomes at 303,000 rps target load:** `availability ≥
  0.995`; `p99Ms < 100` (the tightest bar in this dossier set);
  `costPerMonth ≤ $45,000/mo` (core graph, excluding the optional CDN).

## 11. Ride-Hailing Trip API

**Tags:** write-heavy telemetry, durable ledger · **Difficulty:** Hard

### Frequency evidence
Grokking's curriculum includes "real-time driver and rider matching, geospatial
indexing at scale, surge pricing"; HelloInterview and Exponent both list Uber
in their Hard tier. **Scope note:** the geospatial matching/indexing core of
this problem is explicitly excluded from this dossier (see
[Exclusions](#exclusions), the same reasoning as "Design Google Maps") — this
dossier covers only the **trip-lifecycle API**: request, match (treated as an
opaque downstream call), track, complete, and pay for a trip.

### Problem statement
Design the trip-lifecycle backend for a ride-hailing app: riders request
trips, the (black-box) matching service assigns a nearby driver, the app
tracks trip state via frequent location/status pings from the driver, and
payment settles at trip completion. Scale: **30M DAU** (riders + drivers
combined), sustained average **150,000 rps**, **write-dominated**
(**read:write ≈ 30:70**) — continuous GPS/status pings from active trips
dominate traffic volume, the opposite emphasis from most dossiers in this
set.

### Simulation targets
`global.users` and `rpsPerUser` sized to 150,000 rps total, `readWriteRatio =
0.3` → 45,000 read / 105,000 write.

### Reference architecture
```
users → ratelimiter (per-driver ping cap) → loadbalancer(×2) → server → cache
                                                                   server → queue (trip event stream) → database
```
| Node | Config |
|---|---|
| `ratelimiter` | `limitRps: 140000` — per-client ping-rate cap, protecting the backend from a buggy/chatty client app |
| `loadbalancer` ×2 | `algorithm: least-connections` |
| `server` | `instances: 210, rpsPerInstance: 1000` (210,000 rps cap, ~67% util) |
| `cache` | `hitRatio: 0.95, capacityRps: 300000` — latest known state/location per active trip |
| `queue` | `workers: 1500, jobsPerWorkerRps: 100` (150,000 rps drain) — durable log of every ping/state transition |
| `database` | `shards: 38, readReplicas: 1, maxConnections: 9500` — trip & payment ledger, the system of record |

### Key design insights
- This problem is dominated by continuous **write** traffic (location pings,
  state transitions), the opposite emphasis from most read-heavy staples in
  this set — candidates who reflexively reach for a big read cache without
  first sizing the write/ping path are missing the actual bottleneck.
- The matching algorithm (finding the nearest available driver) is a
  geospatial-indexing problem (quad-tree/geohash/H3) this simulator cannot
  express — scope the design explicitly around the trip-lifecycle state
  machine and ledger, treating "match rider to driver" as an opaque
  downstream call, the same carve-out this dossier set makes for "Design
  Google Maps."
- A per-client rate limiter on location pings protects the backend from a
  single misbehaving client app, independent of overall system load.
- The trip/payment ledger needs the strongest consistency and durability
  guarantees in this graph (money and legally-relevant trip records) — this
  is the one place where "eventually consistent, just cache it" reasoning
  breaks down, which is exactly why it sits behind the async event queue as
  the durable system of record rather than being written directly from every
  ping.
- Splitting "give me the current trip status" (cache-served, high volume,
  loose consistency OK) from "the durable trip/payment record" (queue-then-
  database, strict consistency) mirrors a very common real-world pattern:
  hot ephemeral state vs. durable ledger.

### Common mistakes / nuances
- Putting the matching/geo-search logic *into* this diagram as if it were a
  capacity problem — it's an indexing/algorithm problem, out of scope; call
  it out rather than hand-wave it.
- Under-sizing for continuous ping/write volume — a common mistake is
  sizing for "number of trips" rather than "pings per trip per second,"
  which is 10–50× higher.
- Writing every single location ping synchronously to the durable ledger —
  massively overloads it; pings should flow through the cache and async
  queue, with only trip start/end and payment events needing strong
  durability.
- No rate limiting on the ping endpoint, letting a single misbehaving
  client flood the backend.
- Conflating rider-facing reads (trip status/ETA) with the internal
  driver-location-ingestion pipeline as if they were the same traffic class.

### Grading rubric suggestions
- **Required components:** `loadbalancer` (≥2), `ratelimiter`, `server`,
  `cache`, `queue`, `database`.
- **Topology:** `server` feeds `cache` (reads) and `queue` (writes); `queue`
  feeds `database`.
- **Config:** `queue.workers × jobsPerWorkerRps` sized for the *ping* write
  rps, not just a trip-count estimate; `database.shards` sized so write
  capacity clears queue-drained demand at ≤90% utilization; `cache.hitRatio
  ≥ 0.9`.
- **Simulation outcomes at 150,000 rps target load:** `availability ≥
  0.99`; `p99Ms < 300` (status lookups); `costPerMonth ≤ $110,000/mo`
  (queue worker fleet and DB shard count both driven up by the write-heavy
  ping volume — a direct, expected consequence of this dossier's core
  insight).

## 12. Payment Processing System

**Tags:** idempotency, consistency, ledger design · **Difficulty:** Hard

### Frequency evidence
Alex Xu Vol. 2, ch. 11 ("Payment System") and ch. 12 ("Digital Wallet");
Grokking Vol. II includes a dedicated payment-platform module;
HelloInterview lists Robinhood/payment-system-class problems in its Hard
tier.

### Problem statement
Design a payment processing backend: charge a card / debit a wallet /
transfer funds between accounts, integrating with external payment
providers (card networks, banks), guaranteeing **exactly-once** semantics
(no double charges, no lost payments) even under client or network retries,
maintaining a durable, auditable ledger, and reconciling asynchronously with
providers via webhooks. Scale: **50M DAU**, ~230 transactions/sec average,
designed for a **10× peak** (Black-Friday-class) of ~2,300 tps. Read traffic
(balance checks, transaction history, receipts) numerically dominates actual
money-movement writes (**read:write ≈ 85:15**). Target: **20,000 rps** total.

### Simulation targets
`global.users` and `rpsPerUser` sized to 20,000 rps total, `readWriteRatio =
0.85` → 17,000 read / 3,000 write.

### Reference architecture
```
users → ratelimiter (velocity/fraud) → loadbalancer → server → cache
                                                          server → queue (settlement) → database
```
| Node | Config |
|---|---|
| `ratelimiter` | `limitRps: 19000` — per-account transaction-velocity/fraud safety net, modest relative to load |
| `loadbalancer` | default (utilization stays far below its cap here — this problem's difficulty is consistency, not raw throughput) |
| `server` | `instances: 30, rpsPerInstance: 1000` (30,000 rps cap, ~63% util) |
| `cache` | `hitRatio: 0.8, capacityRps: 300000` — idempotency-key dedup checks and balance-lookup reads |
| `queue` | `workers: 50, jobsPerWorkerRps: 100` — async settlement/reconciliation with the external provider |
| `database` | `shards: 2, readReplicas: 1, maxConnections: 2000` — double-entry ledger, **connection pool deliberately over-provisioned ~4× the bare Little's-Law minimum** for retry-storm resilience |

### Key design insights
- **Idempotency is the single most important correctness property**: every
  payment request must carry a client-generated idempotency key, checked
  (via the cache) *before* any ledger write, so that retries — from the
  client or from your own retrying servers — never cause a double charge.
  This is why the dedup check sits in front of the ledger write path, not
  after.
- The ledger should be **double-entry** (every money movement is a
  debit+credit pair summing to zero) and is the one component in this whole
  dossier set that should be graded *more* on consistency/durability posture
  than on raw throughput headroom — favor more shards/connections/replicas
  than strictly required by load, not fewer.
- Settlement with external providers is inherently asynchronous and
  eventually consistent: tell the user "payment accepted" the instant your
  *own* ledger write succeeds, then reconcile with the provider's
  webhook/callback afterward via the queue — blocking the user-facing
  response on an external round-trip is an availability risk you don't need
  to take.
- Read traffic (balance checks, history, receipts) is a different, far
  higher-volume, far more cache-friendly traffic class than the actual
  money-movement writes — don't let read-path caching reasoning leak into
  how you reason about the strongly-consistent write path.
- Retry storms are a first-class failure mode: a client or flaky network
  retries an already-successful charge. This connects directly to the
  server's own `retriesEnabled`/circuit-breaker knobs and to deliberately
  over-provisioning the database connection pool, since idempotent-but-
  retried requests still consume connections and capacity even when they
  correctly resolve to a no-op.

### Common mistakes / nuances
- No idempotency key / dedup check at all — network retries cause double
  charges, the single most damaging bug class in this domain.
- Performing the ledger write synchronously *inside* the same transaction as
  the call to the external payment provider — ties your availability to
  theirs and complicates the double-entry invariant if the external call
  fails halfway through.
- Treating balance/history reads and payment writes as needing the same
  consistency model — over-applying strict consistency to cache-friendly
  reads wastes capacity; under-applying it to the ledger risks correctness.
- Sizing the database connection pool at the bare average-load minimum
  instead of the burst/retry-storm case — payment systems see retry
  amplification precisely when things are already going wrong.
- No async reconciliation path with the external provider — webhooks/
  callbacks arrive out-of-band and must be handled, not assumed away.

### Grading rubric suggestions
- **Required components:** `ratelimiter`, `loadbalancer`, `server`, `cache`,
  `queue`, `database`.
- **Topology:** `server` feeds `cache` (reads/idempotency-check) and
  `queue` (writes); `queue` feeds `database`; the cache/dedup check must sit
  **before** the ledger write path, not after.
- **Config:** `database.maxConnections` generously above the bare Little's-
  Law minimum for its shard count (an explicit over-provisioning check
  unique to this dossier — reward headroom here rather than penalizing it);
  `database.readReplicas ≥ 1` **and** `shards ≥ 2` (durability/read-scaling
  posture).
- **Simulation outcomes at 20,000 rps target load:** `availability ≥ 0.995`
  (a higher bar than most dossiers — money can't be "mostly available");
  `p99Ms < 250` (user-facing charge confirmation); `costPerMonth ≤
  $8,000/mo` (deliberately modest scale reflecting a "correctness over raw
  throughput" framing, with generous connections/replicas being the main
  deliberate over-spend beyond the bare minimum).

## Exclusions

Top-frequency questions considered and excluded because they are
fundamentally unmodelable in ChaosLab's single-region, non-geo, non-realtime
request-flow simulator — listed here so a downstream agent doesn't
accidentally re-introduce them.

| Question | Sources citing it | Why excluded |
|---|---|---|
| **Google Maps / Proximity Service / Nearby Friends** | Alex Xu Vol. 2 ch. 1–3; Grokking (geospatial indexing at scale) | Core difficulty is geospatial indexing (quad-tree/geohash/H3) and proximity queries — the simulator has no geographic or spatial-indexing semantics at all, only flat RPS capacity. |
| **Google Docs / real-time collaborative editing** | dev.to Hard tier; common Hard-tier follow-up across sources | Core difficulty is operational-transform/CRDT conflict resolution over a live bidirectional connection — no realtime/WebSocket semantics or document-state modeling exists in this engine. |
| **Distributed Key-Value Store / Consistent Hashing design** | Alex Xu Vol. 1 ch. 5–6 | The interesting part of this question (consistent-hashing ring, virtual nodes, live rebalancing on node add/remove) isn't modeled — the simulator's `database.shards` is a static count with no ring topology or rebalancing behavior. |
| **Web Search Engine / ranking** | Common Hard-tier mention (dev.to, Exponent) | Core difficulty is relevance ranking and inverted-index query semantics, not a capacity/flow problem — indistinguishable from the Web Crawler dossier's storage/pipeline concerns without the ranking algorithm, which isn't modelable. |
| **Stock Exchange / Trading Matching Engine** | Alex Xu Vol. 2 ch. 13 | Core difficulty is a deterministic, strictly-ordered order-book matching engine — no matching-engine, ordering-guarantee, or sequencing semantics exist in this engine's capacity model. |
| **Distributed Message Queue (e.g. "Design Kafka")** | Alex Xu Vol. 2 ch. 4 | Asks candidates to design the primitive this simulator already treats as an atomic, opaque component (`queue`) — a recursive/meta problem relative to the catalog, not a graph-composition problem. |
| **Video Conferencing (Zoom-class)** | Common Hard-tier mention (dev.to) | Core difficulty is real-time peer-to-peer/SFU media routing over WebRTC with strict jitter/latency constraints — no realtime media semantics in this engine. |
| **Online Multiplayer Game / Game State Sync (e.g. Online Chess)** | HelloInterview Hard tier | Core difficulty is low-latency bidirectional state synchronization — the same realtime-semantics gap as chat's push/presence layer, but central rather than incidental to this problem. |
| **Ad Click Event Aggregation / Metrics Monitoring** | Alex Xu Vol. 2 ch. 5–6 | Core difficulty is approximate, windowed stream aggregation (HyperLogLog, watermarking, exactly-once counting) — reduces to generic ingest→queue→database in this simulator with none of the aggregation semantics that make the question distinctive. |
